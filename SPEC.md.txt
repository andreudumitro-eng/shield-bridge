1. Introduction
Zcash offers industry‑leading privacy technology through its Orchard shielded pool. However, the Zcash ecosystem remains isolated from the broader cryptocurrency market, resulting in low liquidity and limited adoption. Existing bridges either support only a single asset (wSOL) or require desktop clients, creating friction for mainstream users.

ZecShield is a trust‑minimized bridge that connects Solana and Zcash, allowing any SPL token to be shielded in Zcash's Orchard pool. The protocol uses FROST threshold signatures to eliminate single points of failure and operates entirely through a web interface — no desktop client required.

1.1 Key Principles
Principle	Description
Trust Minimization	No single entity controls the bridge. FROST threshold signatures require a supermajority of signers to authorize any transaction.
Universal Asset Support	Any SPL token on Solana can be shielded, not just a predefined list. This maximizes liquidity potential.
Web‑First Access	Users interact through familiar wallets (Phantom, Backpack) without installing specialized software.
Open Source	All components are publicly available, with contributions back to the Zcash ecosystem.
1.2 Problem Statement
Problem	Impact
Liquidity Isolation	Users cannot easily move assets between Zcash and other chains, limiting utility.
Friction	Existing solutions require desktop clients or complex setup, excluding 95% of potential users.
Limited Asset Support	Existing bridges support only wSOL, leaving billions in other SPL tokens untapped.
ZecShield solves these by creating a low‑friction, high‑security bridge that brings the entire Solana asset ecosystem into Zcash's shielded pools.

1.3 Scope
This document specifies cryptographic primitives, smart contract architecture, orchestrator logic, FROST coordination, Zcash integration, indexer, Proof of Reserves mechanism, security model, deployment roadmap, and governance transition.

2. Bridge Model
ZecShield implements a lock‑and‑mint model with threshold signatures.

2.1 Core Operations
Operation	Direction	Description
Shield	Solana → Zcash	User locks SPL tokens on Solana, receives shielded notes in Zcash Orchard
Unshield	Zcash → Solana	User burns shielded notes, unlocks SPL tokens on Solana
2.2 Trust Assumptions
Component	Trust Required	Rationale
Solana ShieldEscrow	Trust in Solana consensus + audit	Smart contract holds locked assets
Zcash Orchard	Trust in Zcash cryptography	Shielded notes are cryptographically secured
FROST Signers	3-of-5 threshold with economic bonds	No single signer can act alone; malicious behavior results in financial loss
Orchestrator	None	Only coordinates; cannot sign or move funds
Indexer	None	Read‑only; does not affect bridge operations
RPC Providers	Redundant (Helius + QuickNode + public)	No single point of failure

3. Cryptographic Parameters
3.1 FROST (Flexible Round‑Optimized Schnorr Threshold)
Parameter	Value	Rationale
Threshold (t)	3	Requires supermajority but tolerates 2 failures
Signers (n)	5	Geographically distributed: US, EU, APAC, backup
Curve	ed25519	Standard, supported by Zcash Foundation
Library	frost-ed25519	Official Zcash Foundation implementation
Key Generation	Distributed	No single entity ever holds full private key
Signer Bonds	1,000 ZEC (or equivalent SOL)	Economic disincentive for malicious behavior
Slashing Conditions	Equivocation, downtime >7 days, invalid signatures	Active from mainnet launch
3.2 Solana Signatures
Parameter	Value	Rationale
Algorithm	Ed25519	Native to Solana, high performance
Wallet Integration	Phantom, Backpack, Solflare	Most popular Solana wallets
Message Format	Borsh	Standard Solana serialization
3.3 Zcash Orchard
Parameter	Value	Rationale
Shielded Pool	Orchard	Latest Zcash shielded protocol
Address Type	Unified Address (UA)	Supports transparent, Sapling, Orchard
Memo Field	512 bytes	For storing asset metadata
Finality	10 blocks (~10 minutes)	Based on Zcash reorg analysis: 10 blocks provides >99.9% finality
3.4 Argon2id for Share Validation
Parameter	Value	Rationale
Memory	256 MiB	ASIC‑resistance, CPU‑friendly
Iterations	2	Balance: ~100ms on modern CPU
Parallelism	4	Optimal for 4‑8 core CPUs
Version	0x13	Argon2id (hybrid)
Note: Argon2id is used for share validation to prevent spam and ensure fair contribution tracking.

4.2 Escrow Account Creation
For each new SPL token to be supported, a dedicated escrow account is initialized:

Field	Value
Mint	SPL token mint address
Total Locked	0 (initial)
Authority	FROST aggregate public key
Paused	false
Asset ID	SHA256(mint_address + "zecshield")
Proof Commitment	SHA256(asset_id || total_locked || total_shielded || timestamp)
Note: While escrow initialization requires administrative action for security (preventing malicious token spoofing), the bridge is designed to support any SPL token upon request. A permissionless mechanism via DAO vote is planned for Phase 3.

4.3 Token Lock Mechanism
Shield Instruction Logic:

text
1. Verify user ATA has sufficient balance
2. Transfer amount from user ATA to escrow ATA
3. Update escrow.total_locked += amount
4. Emit ShieldEvent with zcash_ua in memo
5. Return success
Invariants:

escrow.total_locked == sum of all locked amounts (enforced by token transfers)

Each shield creates exactly one event

Events are immutable after emission

5.5 Health Checks & Failover
Orchestrator nodes ping each other every 10 seconds

Leader election via etcd (Raft consensus)

If leader fails, backup takes over within 5 seconds

FROST signers ping orchestrator every 30 seconds

Signer offline for >2 minutes → removed from rotation

Minimum 3 signers and 1 orchestrator required for operation

6.2 Key Generation
Keys are generated using distributed key generation (DKG):

text
1. Each signer generates a secret share
2. Signers exchange commitments
3. Signers verify each other's commitments
4. Aggregate public key is computed
5. No single entity ever holds full private key

6.5 Signer Security
Requirement	Implementation
Key Storage	HSM or encrypted filesystem
Network	TLS 1.3 with mutual authentication
Rate Limit	100 requests/minute per signer
Audit Log	All signing attempts logged
Bond Lock	1,000 ZEC locked for 30 days, renewable
6.6 Slashing Mechanism (Active from Mainnet Launch)
Violation	Detection	Penalty
Equivocation (signing two conflicting transactions)	On-chain detection via duplicate signatures	100% bond burned
Invalid Signatures	Signature verification fails	Warning → 50% bond on second offense
Downtime > 7 days	Missed heartbeats	10% bond reduction per additional week
Attempted theft	3-of-5 malicious signers	All bonds burned, signers replaced
Slashing Process:

Violation detected by any observer

Evidence submitted to Solana (special instruction)

FROST signers verify evidence (requires 3-of-5 for execution)

Bond transferred to treasury or burned

8.2 Proof of Reserves
Formula:

text
total_shielded_tokens = Σ(shielded_notes_per_asset)
total_locked_tokens = Σ(escrow.total_locked_per_asset)

Proof holds iff: total_shielded_tokens == total_locked_tokens
On‑Chain Commitment:

Every hour, the orchestrator computes:

text
commitment = SHA256(asset_id || total_locked || total_shielded || timestamp)
This commitment is published on Solana via the update_proof instruction in the ShieldEscrow program. Anyone can verify:

Query Solana escrow account for the latest commitment

Query Zcash indexer for total_shielded

Compare with their own calculation

Verification Process:

text
1. User fetches commitment from Solana escrow
2. User queries Zcash indexer for total_shielded
3. User computes SHA256(asset_id || total_locked || total_shielded || timestamp)
4. If matches → reserves are valid

11. Security
11.1 Threat Model
Threat	Description	Mitigation
Escrow Hack	Vulnerability in ShieldEscrow program	Audit, 3-of-5 pause mechanism, upgradeable proxy
FROST Collusion	3 signers collude to steal funds	3-of-5 threshold + 1,000 ZEC bonds + slashing
Orchestrator DoS	Orchestrator goes offline	Clustered orchestrators with automatic failover (<5s)
RPC Failure	Helius goes down	Redundant RPC pool (Helius, QuickNode, public)
Replay Attack	Same shield event processed twice	Nonce tracking, idempotent operations
UA Spoofing	User sends to wrong address	UA validation, checksum verification
False Slashing	Malicious user submits false evidence	Evidence requires 3-of-5 FROST verification
11.2 Slashing Conditions (Active from Mainnet Launch)
Violation	Evidence	Penalty
Equivocation	Two conflicting signatures from same signer	100% bond burned
Invalid Signatures	Signature fails verification (3+ offenses)	Warning → 50% bond
Downtime	No heartbeat >7 days	10% bond reduction per week
Attempted Theft	3 signers collude	All bonds burned, signers replaced
Slashing Execution:

Any observer can submit evidence to Solana

Evidence verified by FROST signers (3-of-5)

On verification, bond transferred to treasury

16.3 Error Handling
Implementations MUST use checked arithmetic to prevent overflow/underflow.

rust
fn add_locked(escrow: &mut EscrowAccount, amount: u64) -> Result<()> {
    escrow.total_locked = escrow.total_locked
        .checked_add(amount)
        .ok_or(Error::Overflow)?;
    
    if escrow.total_locked > MAX_SUPPLY_LYT {
        return Err(Error::ExceedsMaxSupply);
    }
    
    Ok(())
}
